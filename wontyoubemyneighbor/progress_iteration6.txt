**Issue #1: LLDP Not Working**

Investigation findings:
- lldpd daemon IS running in containers ✓
- lldpd configured correctly via docker-entrypoint.sh ✓
- lldpd interfaces configured (eth0, asi0, etc.) ✓
- **PROBLEM**: lldpd shows "no neighbor"

Root cause:
- Docker bridge networks do NOT forward Layer 2 multicast by default
- LLDP uses multicast MAC address 01:80:c2:00:00:0e
- Frames are being dropped by Docker's Linux bridge
- lldpd running but cannot discover neighbors across bridge

- LLDP Python daemon in agentic/discovery/lldp.py exists ✓

**Fix #1: GRE Tab Data**

File: webui/static/agent-dashboard.js

Added `fetchGREData()` method to fetch from `/api/gre/tunnels`:
```javascript
async fetchGREData() {
    try {
        const response = await fetch('/api/gre/tunnels');
        if (response.ok) {
            const data = await response.json();
            if (data.tunnels && data.tunnels.length > 0) {
                this.protocols.gre = {
                    enabled: true,
                    tunnel_count: data.count,
                    tunnels: data.tunnels
                };
                this.updateGREData(this.protocols.gre);
            }
        }
    } catch (e) {
        console.log('Could not fetch GRE data:', e);
    }
}
```

Modified updateStatus() to call fetchGREData when GRE interfaces detected:
```javascript
if (this.hasGREInterfaces(status.interfaces) &&
    (!this.protocols.gre?.tunnels || this.protocols.gre.tunnels[0]?.local_ip === 'N/A')) {

}
```

Result: Dashboard now fetches GRE data directly from working endpoint

===========================================
LLDP LIMITATION DISCOVERED
===========================================

**Docker Bridge Network Issue**

LLDP requires Layer 2 multicast frame forwarding, but:
- Standard Docker bridge networks filter multicast
- LLDP frames (01:80:c2:00:00:0e) are dropped
- Containers cannot see each other's LLDP advertisements

**Potential Solutions**:

1. **Use macvlan/ipvlan networks** (best for production)
   - Containers get real network interfaces
   - LLDP frames forwarded naturally
   - Requires network configuration changes

2. **Application-level discovery** (current approach)
   - Python LLDP daemon queries lldpd
   - Can supplement with custom discovery
   - Could use routing protocol data
   - Could synthesize neighbors from topology

3. **Bridge configuration** (complex)
   - Modify Docker bridge to forward LLDP
   - Requires iptables/ebtables rules
   - May not work in all environments

**Current Status**:
- lldpd will show "no neighbors" (expected on bridge network)
- Python LLDP daemon will still run
- May need alternative neighbor discovery method
- Could use OSPF/BGP adjacencies as neighbor source
